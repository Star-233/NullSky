
## 用户名枚举

### 提示不同

如果网站把账号不存在和密码错误的提示分开，这意味着账号可以枚举。攻击者可以先枚举用户名，然后再针对爆破。

### 响应时间不同

如果服务端在验证用户名是否存在时耗时不同（例如先查询用户再校验密码，用户存在时多一次数据库查询），可以通过响应时间的差异来判断用户名是否存在。攻击者可以用统计学方法多次请求取平均值来消除网络延迟噪音。

**存在漏洞的代码示例（伪代码）：**

```python
def login(username, password):
    user = db.query("SELECT * FROM users WHERE username = ?", username)
    if not user:
        return "用户名或密码错误"          # 查了一次 DB，快速返回
    if hash(password) != user.password_hash:
        return "用户名或密码错误"          # 多了一次查 DB + 哈希计算，耗时更长
    return "登录成功"
```

用户存在时比不存在时多执行了一次哈希比对，响应时间差异可达几毫秒到几十毫秒。

**安全修复：始终执行相同的操作路径**

```python
def login(username, password):
    user = db.query("SELECT * FROM users WHERE username = ?", username)
    expected_hash = user.password_hash if user else DUMMY_HASH
    if hash(password) != expected_hash:
        return "用户名或密码错误"
    return "登录成功"
```

无论用户是否存在都进行哈希计算，消除时间差异。高级方案还可引入随机延迟（+0~50ms jitter）来进一步模糊时间特征。

### HTTP 状态码不同

服务端对不同的验证结果返回不同的 HTTP 状态码时，可用于枚举用户名。常见场景：
- `200 OK` — 登录成功
- `401 Unauthorized` — 用户名存在但密码错误
- `404 Not Found` — 用户名不存在
- `403 Forbidden` — 账户被锁定或受限
- `400 Bad Request` — 参数格式错误（可与用户名枚举无关）
